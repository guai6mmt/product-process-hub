## 1. 范围与定位

- 全流程为:**产品准入 → 销售准入 → 上架发售 → 存续管理**。本版聚焦**销售准入的"文件门禁"**;其余环节可用同一模型扩展。
- 核心命题:**销售准入能否启动 = 一组文件"备齐且审查通过"**。即"文件门禁"通过,才允许进入下一环。

---

## 2. 核心概念(数据模型,工具无关)

| 实体 | 含义 | 关键关系 |
|---|---|---|
| **产品** Product | 一支在售/拟售产品 | 1 — N 流程实例;N — 1 机构 |
| **流程实例** ProcessInstance | 产品在某环节的一次推进(本版=销售准入) | 隶属 1 产品;1 — N 文件需求 |
| **文件需求** FileRequirement | 清单里的一"格"(要交什么文件) | 由蓝图项实例化;1 — N 文件版本 |
| **文件版本** FileVersion | 某文件需求下的一次上传(带版本号与血缘) | 隶属 1 文件需求;1 — N 审查任务 |
| **审查任务** ReviewTask | 挂在某版本上的审查(类型∈{消保,法审,用印}) | 隶属 1 文件版本 |
| **机构** Institution | 代销渠道/对方机构 | 影响"海报是否纳入""用印对方" |
| **蓝图项** BlueprintItem | 标准清单模板的一行(规则,不是数据) | 实例化 → 生成文件需求 |

> **最关键的一条区分:蓝图(模板/规则)≠ 实例(某产品的数据)。** 蓝图只定义一次;每个新产品由蓝图 + 4 个参数自动"实例化"出自己的清单。这是"不再逐表手动维护"的根基。

## 3. 蓝图:标准文件清单(固定规则)

下表是**所有产品共用的规则**。`✓`=需要,`—`=不需要。

| 项ID | 大类 | 文件项 | 必需性 | 数量规则 | 消保 | 法审 | 用印(方) | 证明表单 | 纳入条件 |
|---|---|---|---|---|:--:|:--:|---|---|---|
| `DX-ZDZ` | 代销协议 | 总对总协议 | 必需 | 固定 1 | ✓ | ✓ | ✓(双方)³ | 流程表单·选传 | 总是 |
| `DX-BC` | 代销协议 | 代销协议补充协议 | 必需 | **1~N**(实例填 N) | ✓ | ✓ | ✓(双方)³ | 流程表单·选传 | 总是 |
| `WT-HAN` | 委托代销函 | 委托代销函 | 必需 | 固定 1 | — | — | ✓(对方机构) | 以"是否用印"标记 | 总是 |
| `XC-PPT` | 宣传材料 | PPT | 必需 | 固定 1 | ✓ | — | — | 消保表单·**必传** | 总是 |
| `XC-YYT` | 宣传材料 | 一页通 | 必需 | 固定 1 | ✓ | — | — | 消保表单·**必传** | 总是 |
| `XC-HB` | 宣传材料 | 海报 | **条件必需** | 条件 0/1 | ✓ | — | — | 消保表单·**必传** | 机构有海报时 |
| `XC-SL` | 宣传材料 | 双录话术 | 必需 | 固定 1 | ✓ | — | — | 消保表单·**必传** | 总是 |
| `XC-ZB` | 宣传材料 | 线上智能播报 | 必需 | 固定 1 | ✓ | — | — | 消保表单·**必传** | 总是 |
| `HT-GRP` | 合同 | **合同文件组**(子件见 §4) | 必需 | **可配置子件 1~N** | ✓ | ✓ | — | 选传 | 总是;子件按形态生成 |

³ **用印方**:代销协议 / 补充协议 = 双方(我方 + 对方机构);委托代销函 = 对方机构。(若实际仅需单方,告知即可改;见 §0 / A1。)

**关于「合同文件组」:** 合同不是固定 1 行或 3 行,而是一个**可配置文件组**——实例化时按"合同形态"选定若干**子件**,每个子件**独立**走消保 + 法审、各自为必需项。子件目录与形态预设见 §4 / P4。这样既支持"合并 1 份 / 标准拆分 3 份",也支持未来某机构"拆成 2 部分"等自定义。

## 4. 实例化:4 个可变参数 → 自动生成清单

**每个新产品只需提供 4 个参数**,系统据此把蓝图展开成该产品的文件需求清单:

| 参数 | 取值 | 作用 |
|---|---|---|
| **P1 机构** | 选择一个机构 | 决定 P3 默认值、各项用印对方 |
| **P2 补充协议份数 N** | 整数 ≥ 1 | 生成 N 行 `DX-BC`(补充协议 #1…#N) |
| **P3 海报是否纳入** | 是 / 否(默认随机构) | 是→纳入 `XC-HB`;否→排除 |
| **P4 合同形态** | 预设 `合并(1件)` / `标准拆分(3件)` / `自定义(N件)` | 选定合同子件;每件独立消保 + 法审 |

**合同子件目录(可扩展):** 合同主体、风险揭示书、计划说明书(未来可增项)。
**合同形态预设:**
- `合并(1件)`:1 件 = 合同(含风险揭示书 + 计划说明书)
- `标准拆分(3件)`:合同主体 + 风险揭示书 + 计划说明书
- `自定义(N件)`:任意组合(如某机构拆成 2 件:合同主体 + 风险揭示书&计划说明书合订)

**生成算法(确定性):**

```
对蓝图每一项 item:
  若 item.纳入条件 不满足(P3 / 机构) → 跳过
  否则按 item.数量规则 展开:
    固定1 / 条件0或1   → 生成 1 行文件需求
    1~N(DX-BC)        → 生成 N 行(N = P2)
    合同组(HT-GRP)     → 按 P4 形态生成对应子件,每件均为必需且走消保 + 法审
  每行文件需求继承 item 的:大类、名称、必需性、适用审查集合(含用印方)、证明表单要求
```

> 这一步把"手动逐行搭表"变成"答 4 个问题"。生成后,你只维护"原子事实"(见 §6),其余全部派生。

## 5. 审查状态机(消保 / 法审 / 用印)

每个审查任务是挂在**文件版本**上的独立子状态机。

**状态枚举:**

| 审查类型 | 状态 |
|---|---|
| 消保 / 法审 | `未启动` → `送审中` → `通过` / `退回` |
| 用印 | `待用印` → `已用印`(可选 `不适用`) |

**用印方:** 代销协议 / 补充协议 = 双方(我方 + 对方机构);委托代销函 = 对方机构。其余文件无用印。

**规则:**

1. **退回不覆盖,派生新版本。** 审查 `退回` 后,按意见修改 → 上传为**新版本**(版本号 +1,记录"由上一版退回派生"),旧版本留痕。这是版本血缘的来源。
2. **顺序守卫(软约束):消保 → 法审 → 用印。** 系统不硬拦截(飞书也做不到),而是体现在"下一步动作"里:消保未过时不提示送法审。(委托代销函无消保/法审,直接进入用印。)
3. **适用性。** 仅对该文件"适用"的审查才计入(见 §3)。如 `XC-PPT` 只有消保;`WT-HAN` 只有用印。

## 6. 派生逻辑(自动计算 — 整套系统的驱动核心)

你只手动维护 4 类**原子事实**:`是否已上传` / `消保状态` / `法审状态` / `用印状态`。其余全部自动算。

### 6.1 单个文件需求的状态
- **未开始**:无任何已上传版本。
- **进行中**:已上传,但下方"下一步动作"未走到"已到位"。
- **已到位 ✓**:已上传 **且** 所有适用审查=通过/已用印 **且**(证明表单若必传则已上传)。

### 6.2 单个文件的「下一步动作」(有序判定,取第一条命中)
1. 无有效版本 → **「上传〔文件项〕」**
2. 适用消保 且 消保≠通过 →
   `未启动`:**「送消保」** ／ `送审中`:**「等消保结果/催办」** ／ `退回`:**「按消保意见改后重传(新版本)」**
3. 适用法审 且 法审≠通过 → 同上(送/等/改)**法审**
4. 适用用印 且 用印≠已用印 → **「推动〔用印方〕用印」**(对方机构 / 双方)
5. 证明表单=必传 且 未上传 → **「上传审查表单」**
6. 以上都满足 → **「已到位 ✓」**

### 6.3 产品「是否备齐」
`备齐 = 所有"必需"文件需求的状态都 = 已到位`。
(条件项如海报:机构无海报→不纳入,不计入;一旦纳入则视为必需。)

### 6.4 进度
`进度% = 已到位的必需项数 ÷ 必需项总数`。

### 6.5 销售准入门禁
`允许进入"上架发售" = 该产品销售准入实例.备齐 == true`。

## 8. 飞书落地映射(当前实现)

| 本规格概念 | 飞书承载 |
|---|---|
| 蓝图项(§3) | 表④ **文件需求模板** |
| 文件需求(实例) | 表② **文件需求** |
| 文件版本 | 表③ **文件版本库** |
| 产品 | 表① **产品台账** |
| 机构 | 新增「机构」表(供 P1 / 海报 / 用印对方引用) |
| 审查任务 | 审查状态记在表② 文件需求的「消保审查 / 法审审查 / 机构用印」(当前版本的状态);历史版本与「退回意见」留在表③ 文件版本库 |
| 是否备齐 / 下一步动作 / 进度 | 表①、表② 的**公式字段**(§6 逻辑) |
| **实例化(§4)** | **「一键开产品」**:按钮字段 + 自动化 / 脚本,按 4 参数从表④ 生成表② 的行 |
| 驾驶舱 | 表② 的分组视图(按产品分组、筛"未到位")+ 仪表盘 |

> 落地注意:① 顺序守卫只能软提示;② **用印现适用 3 类文件**(总对总 / 补充协议 / 委托代销函),"下一步动作""是否备齐"公式需对这些行计入用印状态;③ "按参数批量生成带关联的行"是飞书原生自动化的弱项,是本阶段要补的主要能力(见后续实现方案)。
